# ソフトウェア開発 GitHub管理ルール

## 1. 目的

本ドキュメントは、旭光電機におけるソフトウェア開発のバージョン管理、修正履歴の把握、リリース管理を適切に行うためのGitHub利用ルールを定めるものです。

## 2. 基本方針

今後、ソフトウェア開発を行う場合は、必ずGitHubにソフトウェアを登録し、以下を実施すること。

- **バージョン管理**: コードの変更履歴を適切に管理
- **修正履歴の把握**: コミットメッセージによる変更内容の記録
- **リリースのタグ付け**: リリースバージョンの明確な管理

## 3. オーガニゼーション

### 3.1 使用するオーガニゼーション

旭光電機のGitHubオーガニゼーション: **`kyokko-electric`**

### 3.2 オーガニゼーション管理者

- **管理者**: 永井
- **役割**: オーガニゼーションのメンバー管理、権限設定、全体の運用管理

### 3.3 オーガニゼーションへの登録手順

1. GitHubアカウントを作成（未作成の場合）
2. 永井に以下の情報を連絡し、オーガニゼーションへの登録を依頼
   - GitHubアカウント名（ユーザー名）
   - 氏名
   - 担当プロジェクト
3. 永井から招待メールが届いたら、承認して登録完了

## 4. リポジトリ作成・管理ルール

### 4.1 リポジトリの作成

- リポジトリは **`kyokko-electric`** オーガニゼーション配下に作成すること
- リポジトリ名は、プロジェクト名や製品名を反映した分かりやすい名前を付けること
- リポジトリの可視性（Public/Private）はプロジェクトの性質に応じて適切に設定すること
  - 原則として **Private** を推奨

### 4.2 必須ファイル

各リポジトリには以下のファイルを必ず含めること:

- **README.md**: プロジェクトの概要、セットアップ方法、使用方法
- **.gitignore**: ビルド成果物、一時ファイル等の除外設定
- **LICENSE** (必要に応じて): ライセンス情報

## 5. ブランチ管理

### 5.1 ブランチ戦略

- **main** (または **master**): リリース可能な安定版コード
- **develop**: 開発中のコード（任意）
- **feature/xxx**: 機能追加用ブランチ
- **bugfix/xxx**: バグ修正用ブランチ
- **hotfix/xxx**: 緊急修正用ブランチ

### 5.2 ブランチ運用

- mainブランチへの直接コミットは原則禁止
- 機能追加やバグ修正は専用ブランチを作成して作業
- mainブランチへの統合はレビュー後に実施（プルリクエスト推奨）

## 6. コミット規則

### 6.1 コミットメッセージ

コミットメッセージは以下の形式で記述すること:

```
[種別] 簡潔な変更内容の要約

詳細な変更内容の説明（必要に応じて）
```

**種別の例:**
- `[機能追加]` または `[feat]`: 新機能の追加
- `[バグ修正]` または `[fix]`: バグの修正
- `[リファクタリング]` または `[refactor]`: コードの整理・改善
- `[ドキュメント]` または `[docs]`: ドキュメントの更新
- `[テスト]` または `[test]`: テストコードの追加・修正

**例:**
```
[機能追加] LED点滅機能の実装

GPIO制御によるLED点滅機能を追加
点滅間隔は500msに設定
```

### 6.2 コミット頻度

- 意味のある単位でこまめにコミットすること
- 複数の変更を1つのコミットにまとめすぎないこと

## 7. リリース管理

### 7.1 バージョニング

セマンティックバージョニング（Semantic Versioning）に従うこと:

```
v<MAJOR>.<MINOR>.<PATCH>
```

- **MAJOR**: 互換性のない大きな変更
- **MINOR**: 後方互換性のある機能追加
- **PATCH**: 後方互換性のあるバグ修正

**例:** `v1.0.0`, `v1.2.3`, `v2.0.0`

### 7.2 タグ付け

リリース時には必ずGitタグを付けること:

```bash
git tag -a v1.0.0 -m "リリースバージョン 1.0.0"
git push origin v1.0.0
```

### 7.3 リリースノート

GitHubのRelease機能を使用して、リリースノートを作成すること:

- 変更内容のサマリー
- 新機能の説明
- バグ修正の一覧
- 既知の問題（あれば）

## 8. 管理対象外ファイル

以下のファイルはGitHubにコミットしないこと（.gitignoreに追加）:

- ビルド成果物（.bin, .hex, .elf等）
- 一時ファイル、キャッシュファイル
- IDEの設定ファイル（プロジェクト固有でないもの）
- 機密情報（パスワード、APIキー等）
- 大容量のバイナリファイル

## 9. セキュリティ

- ソースコードに機密情報（パスワード、APIキー、秘密鍵等）を含めないこと
- 万が一コミットしてしまった場合は、直ちに管理者（永井）に報告すること

## 10. 問い合わせ先

GitHubの運用に関する質問、トラブル、オーガニゼーション登録依頼は以下に連絡すること:

- **担当者**: 永井
- **対応内容**:
  - オーガニゼーションへの登録・削除
  - リポジトリの権限設定
  - GitHub運用に関する相談
